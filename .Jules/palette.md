## 2024-05-24 - Accessibility Fallbacks for Compact Forms Without Explicit Labels
**Learning:** The application heavily relies on "compact" UI designs where form inputs (like search bars, quick-reply textareas, and simple price inputs) use only `placeholder` attributes instead of explicit, visible `<label>` elements. While aesthetically clean, `placeholder` is not reliably announced by all screen readers as an accessible name, leaving visually impaired users without context. This is a recurring pattern across the public tracking pages and admin dashboards.
**Action:** When working on this repository, strictly enforce the use of `aria-label` attributes on any `input`, `textarea`, or `select` element that lacks an associated `<label>` to ensure baseline accessibility without breaking the intended compact visual design.

## 2024-05-25 - Global Form Submission State for Jinja/Tailwind Apps
**Learning:** In purely server-rendered applications without a frontend framework (like this FastAPI+Jinja2 app), it's easy to forget loading states, leaving users unsure if a click registered. Injecting a global script into `base.html` that listens for `submit` events and modifies the primary button (`button:not([type="button"])`) provides a massive UX win with almost zero per-page effort. The approach of using a brief `setTimeout(..., 0)` to allow the actual form submission to proceed before disabling the button and setting an `aria-live` text ensures both functionality and accessibility are preserved.
**Action:** Use this global interception pattern (`document.addEventListener("submit", ...)`) in server-rendered templates to apply consistent disabled states, loading spinners, and screen reader announcements without needing to touch individual forms, but always provide an opt-out class (e.g., `no-global-submit`) for special cases.
